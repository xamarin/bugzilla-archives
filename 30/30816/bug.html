Description Michael Stonis 2015-06-04 15:04:47 UTC 
Our client has an application that will infrequently crash on 64-bit devices. The crash happens when a new ViewController is being pushed to a screen. This navigation is being driven by a storyboard. This same crash scenario has happened on a few screens in the application.

The only consistency that we have seen is from the following two lines

4   libsystem_malloc.dylib        	0x0000000196fb52f4 nano_free + 188
5   CoreFoundation                	0x0000000184e95670 CFRelease + 1060

Each of these crashes comes after a CFRelease and nano_free.

The application has been in production for a few years and did not exhibit this behavior before unified and still does not exhibit this behavior on 32-bit systems.

When this crash occurs, there is no log in Xamarin.Insights and it does not bubble up an exception through the stack.

This looks like memory corruption of some sort, and the randomness can make it tricky to track down.

The first step is to create a reproducible test case. In these cases where the crash only happens once in a while, my suggestion is to automate the app somehow. In your case you could try adding code that enters a view controller that crashes and then a couple of seconds later returns, and loop every 5s. Then leave the app running overnight, and see if it crashes - if not, then you need to find some other code sequence to iterate over. Once you have something that reliably crashes overnight, then remove chunks code from your project until you've created a minimal test project.

In my experience once you have a minimal test project, you end up understanding what's happening. If not, feel free to attach the project here, and we'll have a look.

Tip: it may be that the crash depends on the GC running at a specific time. In that case try adding the following code somewhere in the startup path:

>new Thread (() =>
>{
>    while (true) {
>        Thread.Sleep (10000);
>        GC.Collect ();
>    }
>}).Start ();

and see if the crash is more or less frequent (it may also be that the crash depends on high memory usage and/or memory warnings, in which case the above code snippet will make the crash less likely).
