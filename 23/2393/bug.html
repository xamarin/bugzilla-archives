Comment 13 Kristian Rietveld (inactive) 2012-01-28 05:19:51 UTC 
My brief analysis of this bug.  The crash occurs when Pango is looking for the best font face for the requested font.  During this process, the CoreText backend has to translate CTFontDescriptors to PangoFontDescriptions. It appears that there are some fonts which do not have a font family name and/or style name set. It is clear that we have to guard for such broken fonts, this will also fix the crash that's shown in the back traces.

I am not sure how this relates to the crash exposing itself when the About dialog in Mono Develop is opened.  It is well possible that when the About Dialog is opened and the font "Tahoma" is requested for rendering, some of the broken fonts on a system are being iterated during the best font matching. However, I cannot confirm this. As has been noted before, this very much depends on the set of installed fonts on a user's system.

CoreText can return NULL for style name on both Lion and Snow Leopard. However, a NULL font family name only occurs on Snow Leopard, the version of CoreText shipped with Lion appears to include some magic to return the correct  font family name after all.  In our backend code, we correct this problem by querying the font name (postscript name) when we get a NULL font family name. From the postscript name we determine the family name manually.

We have also put in a guard for a NULL bitmap when requesting font coverage.

I have tested this with the font "Skia" which is installed on Lion and Snow Leopard by default. CoreText is not returning a style name and font family name for this font on Snow Leopard. On Lion the Oblique renditions of this font (which I guess are synthesized by the backend) do not seem to work, but they do work on Snow Leopard (the irony). Perhaps determining the family name from the postscript name works better than the font family names returned by CoreText...
