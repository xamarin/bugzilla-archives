This is a tough nut to crack. Jeff is coordinating with Lluis to try and get more info about this kind of crash. Here are some instructions from them about a new tool for figuring this out:

Lluis wrote a plugin for Mono that detects uses of Gtk from outside of the UI thread called GUI Thread Check which you can find on GitHub: https://github.com/slluis/gui-thread-check

To install it:

$ git clone git@github.com:ssluis/gui-thread-check.git
$ cd gui-thread-check
$ CFLAGS=-m32 ./autogen.sh --prefix=$MONO_INSTALL_PREFIX && make && make install

To use it from within your monodevelop git repository:

$ $MONO_INSTALL_PREFIX/bin/mono --profile=gui-thread-check main/build/bin/MonoDevelop.exe


This should log messages like: *** GTK CALL NOT IN GUI THREAD: ...

If you get any messages like that, let me know what the backtrace is so I can fix those instances

So what was happening here is that the we were saving documents in the build thread instead of the main (UI) thread, and the Document object was firing off events that it had been saved, which then updated the UI and caused these crashes.

I never saw this myself when running the tool because I've developed the habit of religiously saving files I'm editing (probably formed from my days of using NEdit).
